# Buenas prácticas y calidad del software

## codigo-limpio-y-legibilidad

El código se lee muchas más veces de las que se escribe, por lo que la legibilidad es una virtud primordial. Robert C. Martin, en su libro *Clean Code*, sistematizó principios para escribir código que humanos puedan entender con facilidad. El primero es el uso de nombres significativos: variables, funciones y clases deben revelar su intención sin necesidad de comentarios. `diasDesdeUltimaActualizacion` comunica más que `d`. Las funciones deben ser cortas, hacer una sola cosa y operar en un único nivel de abstracción, evitando efectos secundarios sorpresivos.

La consistencia en el estilo —indentación, convenciones de nomenclatura como camelCase o snake_case, y formato— reduce la carga cognitiva. Las guías de estilo, como la PEP 8 de Python, estandarizan estas decisiones. Los comentarios deben explicar el "porqué", no el "qué", pues un buen código es autoexplicativo; un comentario que describe lo obvio es ruido, y uno desactualizado es engañoso. Principios complementarios incluyen DRY (Don't Repeat Yourself), que combate la duplicación, KISS (Keep It Simple, Stupid), que favorece soluciones simples, y YAGNI (You Aren't Gonna Need It), que desalienta añadir funcionalidad especulativa. La reducción de la complejidad ciclomática —el número de caminos posibles a través del código— facilita las pruebas y el mantenimiento. El refactoring, término popularizado por Martin Fowler, es la mejora continua de la estructura interna sin alterar el comportamiento externo, manteniendo el código sano a lo largo del tiempo.

## principios-solid-y-diseno

SOLID es un acrónimo de cinco principios de diseño orientado a objetos formulados por Robert C. Martin que guían hacia sistemas flexibles, mantenibles y extensibles. El principio de Responsabilidad Única (Single Responsibility) sostiene que cada clase debe tener una sola razón para cambiar, encapsulando una única responsabilidad. El principio Abierto/Cerrado (Open/Closed), de Bertrand Meyer, indica que las entidades deben estar abiertas a la extensión pero cerradas a la modificación: se añade comportamiento nuevo sin tocar el código existente, típicamente mediante herencia o composición.

El principio de Sustitución de Liskov, de Barbara Liskov, exige que los objetos de una subclase puedan reemplazar a los de su superclase sin alterar la corrección del programa, garantizando jerarquías coherentes. El principio de Segregación de Interfaces recomienda interfaces pequeñas y específicas en lugar de una grande que obligue a implementar métodos innecesarios. El principio de Inversión de Dependencias propone que los módulos de alto nivel no dependan de los de bajo nivel, sino ambos de abstracciones, reduciendo el acoplamiento. Más allá de SOLID, el buen diseño busca alta cohesión (los elementos de un módulo están fuertemente relacionados) y bajo acoplamiento (los módulos dependen poco entre sí). Los patrones de diseño de la Gang of Four —como Factory, Observer, Strategy y Singleton— ofrecen soluciones reutilizables a problemas recurrentes, aunque deben aplicarse con criterio para no sobrediseñar.

## pruebas-y-testing

Las pruebas son la red de seguridad que permite modificar software con confianza. Verifican que el código se comporta como se espera y detectan regresiones cuando se introducen cambios. Existen varios niveles. Las pruebas unitarias examinan la unidad más pequeña —una función o método— en aislamiento, usando frameworks como pytest, JUnit o Jest. Las pruebas de integración verifican que varios componentes funcionen juntos correctamente. Las pruebas de extremo a extremo (end-to-end) simulan el comportamiento completo del usuario a través de todo el sistema. La pirámide de pruebas recomienda muchas pruebas unitarias rápidas, menos de integración y pocas end-to-end, que son lentas y frágiles.

El desarrollo guiado por pruebas (TDD), promovido por Kent Beck, invierte el orden habitual: primero se escribe una prueba que falla, luego el código mínimo para pasarla, y finalmente se refactoriza. Este ciclo "rojo-verde-refactor" produce diseños más limpios y cobertura desde el inicio. La cobertura de código mide qué porcentaje se ejecuta durante las pruebas, aunque una cobertura alta no garantiza ausencia de errores. Técnicas complementarias incluyen los mocks y stubs para aislar dependencias, las pruebas de propiedades que generan entradas aleatorias, y la integración continua (CI), que ejecuta automáticamente toda la batería de pruebas en cada cambio. Edsger Dijkstra advirtió que "las pruebas pueden demostrar la presencia de errores, pero nunca su ausencia", recordando los límites del testing frente a la verificación formal.

## control-de-versiones-git

El control de versiones registra la historia de cambios de un proyecto, permitiendo colaborar, revertir errores y entender la evolución del código. Git, creado por Linus Torvalds en 2005 para el desarrollo del núcleo Linux, es el sistema dominante. Es distribuido: cada desarrollador posee una copia completa del repositorio con todo su historial, lo que permite trabajar sin conexión y aporta robustez frente a fallos centralizados, a diferencia de sistemas anteriores como CVS o Subversion.

El flujo básico gira en torno a tres áreas: el directorio de trabajo, el área de preparación (staging) y el repositorio. Los cambios se preparan con `git add` y se confirman con `git commit`, creando una instantánea identificada por un hash criptográfico. Las ramas (branches) son el corazón de Git: permiten desarrollar funcionalidades en paralelo de forma aislada y fusionarlas (merge) después, o reorganizar la historia con rebase. Plataformas como GitHub, GitLab y Bitbucket añaden colaboración mediante pull requests o merge requests, revisión de código y automatización. Buenas prácticas incluyen commits pequeños y atómicos con mensajes descriptivos, ramas de corta vida y estrategias de ramificación como Git Flow o trunk-based development. La resolución de conflictos, cuando dos personas modifican las mismas líneas, es una habilidad esencial. El control de versiones es además la base de la integración y entrega continuas modernas.

## manejo-de-errores-y-seguridad

El manejo robusto de errores distingue al software profesional del frágil. En lugar de ignorar fallos o dejar que el programa colapse, el código debe anticipar condiciones excepcionales y responder con elegancia. El mecanismo de excepciones (try/catch o try/except) separa la lógica normal del manejo de errores, propagando fallos hasta donde puedan tratarse adecuadamente. Es importante capturar excepciones específicas en lugar de genéricas, liberar recursos con bloques `finally` o gestores de contexto, y no silenciar errores sin registrarlos. Algunos lenguajes modernos como Rust y Go prefieren devolver valores de error explícitos (Result, error) en vez de excepciones, forzando al programador a manejarlos.

La seguridad debe integrarse desde el diseño, no añadirse al final. La regla de oro es nunca confiar en la entrada del usuario: toda entrada debe validarse y sanearse. Vulnerabilidades clásicas catalogadas por el proyecto OWASP incluyen la inyección SQL, que se previene con consultas parametrizadas; el cross-site scripting (XSS), mitigado escapando la salida; y el desbordamiento de búfer en lenguajes de bajo nivel. El principio de mínimo privilegio limita los permisos al mínimo necesario. Las contraseñas deben almacenarse con funciones de hash robustas y saladas como bcrypt o Argon2, nunca en texto plano. La gestión de secretos, el cifrado de datos sensibles en tránsito (TLS) y en reposo, y las actualizaciones de dependencias para corregir vulnerabilidades conocidas completan una postura defensiva. La seguridad es un proceso continuo, no un estado final.

Conexiones: la calidad del software se apoya en la lógica formal (verificación, aserciones, la advertencia de Dijkstra sobre las pruebas) y en las matemáticas (funciones hash criptográficas, complejidad ciclomática); las buenas prácticas reflejan principios epistemológicos sobre la claridad y la comunicación del conocimiento; la colaboración con Git y la revisión de código son fenómenos sociológicos de trabajo coordinado; y la seguridad conecta con la ética y la teoría de la confianza.
